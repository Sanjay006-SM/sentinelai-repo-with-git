# Stage 5: AI Decision Intelligence Engine & Stage 6: Explainability Engine

This directory contains the automated intelligence capability embedded within SentinelAI. It utilizes the Gemini 2.5 Flash Large Language Model to interpret, contextualize, and summarize complex security events for both executive leadership and technical SOC analysts. 

---

## 1. Problem Solved

When a machine identity is flagged with a high risk score, SOC analysts typically spend hours manually gathering context across different tools—correlating raw logs with Graph database relationships. 

This service completely automates the context-gathering phase. **Crucially, the AI does not calculate risk itself.** Instead, it interprets the verified risk evidence generated by the Risk Intelligence Engine (Stage 4) and converts it into meaningful insights, remediation recommendations, executive summaries, compliance guidance, and conversational responses via Retrieval-Augmented Generation (RAG).

---

## 2. Architecture & Pipeline Alignment

The feature is built on an orchestration-and-collection architecture, isolating the API calls from data gathering to satisfy Stage 5 and Stage 6 requirements.

* **Stage 5 (AI Decision Intelligence Engine)**:
  * **`AIAnalystService`** (`ai_analyst_service.py`): The core reasoning layer. It wraps the `google.genai` SDK, managing retries, payload configuration, and output parsing to generate explanations from deterministic evidence.
  * **`PromptBuilder`** (`prompt_builder.py`): Compiles gathered telemetry into a strict, constrained prompt template demanding JSON output.
* **Stage 6 (Explainability Engine)**:
  * **`EvidenceCollector`** (`evidence_collector.py`): Acts as the trust layer. It retrieves and attaches the exact evidence (PostgreSQL logs, Neo4j attack paths, prior risk scores) to validate that every statement generated by the AI is supported by the underlying Enterprise Knowledge Graph.

---

## 3. Execution Pipeline

The engines execute via the following deterministic pipeline:

**Pipeline Step 1: Evidence Collection (Stage 6 Trust Layer Base)**
The `EvidenceCollector` extracts a multi-dimensional snapshot of the identity from PostgreSQL and Neo4j. This forms the verifiable truth that grounds the AI and prevents hallucination.

**Pipeline Step 2: Token Optimization & Prompt Construction**
To remain within context windows and optimize inference latency, the `PromptBuilder` formats the evidence into a highly constrained prompt enforcing strict JSON output schemas.

**Pipeline Step 3: LLM Inference with Resiliency (Stage 5)**
The `AIAnalystService` submits the prompt to Gemini 2.5 Flash via the `generate_content` SDK method, strictly defining the `response_mime_type="application/json"` in the API configuration. 

**Pipeline Step 4: Validation & Parsing (Explainability Check)**
The raw string output is parsed into a Python dictionary. The service statically enforces the presence of required schema keys (`executive_summary`, `risk_assessment`, `attack_path_analysis`, `findings`, `recommendations`) to guarantee API contract stability before passing it upstream.

---

## 4. Major Features & Resilience

### Constrained JSON Prompting & Key Verification
* **Purpose**: Guarantee programmatic usability of the LLM output and prevent API crashes due to missing nodes.
* **Implementation**: The `PromptBuilder` instructs the LLM to skip markdown formatting and return raw JSON. Missing string/array keys are safely initialized to empty defaults to maintain the downstream contract.

### Reliability & Resiliency
* **Automatic Retries & Exponential Backoff**: The LLM call is protected by the `tenacity` library, retrying on any exception up to 3 times with exponential backoff.
* **Token Guardrails**: The service explicitly limits `recent_activity` logs to a maximum of 20 ordered by descending time (`limit(20)`) to prevent context window bloat and reduce cost.

### Explainability / Observability
* **Explicit Prompt Directives**: The LLM is given strict anti-hallucination commands ("BASE ALL FINDINGS STRICTLY ON EVIDENCE", "DO NOT ASSUME PERMISSIONS"). 
* **Database Aggregation**: Instead of dumping thousands of raw logs into the prompt, the `EvidenceCollector` performs heavy SQL aggregation locally and only feeds synthesized integers into the LLM.
