# Stage 1: Enterprise Data Ingestion Engine & Stage 4: Risk Intelligence Engine

This directory contains the core analytical and orchestration components responsible for continuously evaluating the security posture and behavior of machine identities (e.g., AWS IAM Roles, Users, and Services). By utilizing a hybrid data architecture (Graph + Relational), the engine computes a dynamic risk score (0-100) and generates actionable security findings based on behavioral heuristics.

---

## 1. Problem Solved

Modern cloud environments generate millions of access logs. Identifying a compromised or overly-permissive machine identity is difficult because threats often span multiple dimensions: relational volume (e.g., excessive API calls) and complex relationships (e.g., cross-account role assumption). 

Traditional implementations rely solely on relational SIEMs or static threshold alerts, which fail to capture relationship-based lateral movement. This implementation solves the problem by running concurrent heuristics across a **Neo4j Graph Database** (for traversal and relationship anomalies) and a **PostgreSQL Database** (for volume, time-series, and payload analysis).

---

## 2. Architecture & Architectural Alignment

The architecture strictly separates orchestration from heuristic calculation, adhering to the SentinelAI pipeline definitions:

* **Stage 1 (Ingestion)**: The `IngestionService` (`ingestion.py`) acts as the entry point into the organization’s cybersecurity ecosystem. It continuously collects security telemetry, parses it (via `CloudTrailParser`), safely resolves conflicts using Postgres-native `ON CONFLICT DO NOTHING`, and packages validated events for normalization.
* **Stage 4 (Risk Intelligence)**: The `RiskEngine` (`risk_engine.py`) and `RiskFactorCalculator` (`risk_factor_calculator.py`) form the analytical core. They perform context-aware graph analysis to determine how cyber threats could propagate throughout the organization, rather than evaluating assets individually.
* **Database Models** (Located in `app/models/`):
  * `MachineIdentity`: Denormalized entity representing the target.
  * `RiskScore`: Snapshot of the aggregate 0-100 score and the list of reasons.
  * `RiskFinding`: Discrete, queryable security events generated by the heuristics.
  * `AccessLog`: The raw time-series data feed containing JSONB payloads.

---

## 3. Execution Pipeline

The execution flow operates in the following stages:

**Pipeline Step 1: Log Ingestion & Conflict Resolution (Stage 1)**
`IngestionService` parses raw logs from multiple heterogeneous sources. It safely upserts into the `AccessLog` table to prevent duplicate events.

**Pipeline Step 2: Context Loading**
The `RiskEngine` receives a target `MachineIdentity`. This can occur globally (`evaluate_all`) or incrementally during log ingestion for only active identities (`evaluate_new_identities`).

**Pipeline Step 3: Hybrid Heuristic Evaluation (Stage 4)**
The orchestrator triggers the `RiskFactorCalculator` to run multiple discrete rules. Each rule queries either Neo4j or PostgreSQL depending on the access pattern required, returning a localized score penalty and a descriptive reason.

**Pipeline Step 4: Normalization & Severity Assignment**
The cumulative penalties are aggregated and strictly bounded to a 0-100 range. Based on the final score, a string severity is assigned (`Critical`, `High`, `Medium`, or `Low`).

**Pipeline Step 5: Persistence & Handoff**
New `RiskFinding` and `RiskScore` records are instantiated. The total score is denormalized directly onto the `MachineIdentity` model. This verified risk intelligence is then made available to the AI Decision Intelligence Engine for the next pipeline stage.

## 4. Major Features

### Privilege Escalation Detection
* **Implementation**: Issues a Neo4j Cypher query traversing the `ASSUMED_ROLE` relationship to track identity chaining and blast radius.

### Sensitive Resource Access
* **Implementation**: Groups PostgreSQL `AccessLog` records by `event_source`. Flags interactions with KMS, IAM, SecretsManager, and sensitive S3 bucket actions.

### Geographic Anomaly Detection
* **Implementation**: Queries the Neo4j graph for `ORIGINATED_FROM` edges to detect distributed or anomalous origin IPs.

### Failed API Call Tracking
* **Implementation**: Utilizes PostgreSQL JSONB querying (`raw_event_json.op('->>')('errorCode')`) to identify and count `AccessDenied` or `UnauthorizedOperation` errors.

### Dormant Then Active Detection
* **Implementation**: Analyzes the delta between `first_seen` and `last_seen` timestamps to detect credential hijacking of abandoned identities.

---

## 5. Reliability, Security & Governance

* **Idempotent Evaluations & Constraints**: The engine ensures idempotent updates by explicitly clearing existing `RiskScore` and `RiskFinding` rows. Score boundaries (0-100) are enforced via PostgreSQL `CheckConstraint`.
* **Workspace Isolation**: Multi-tenant data segregation is strictly enforced via `workspace_id` filtering in both Neo4j Cypher parameters and SQLAlchemy where clauses.
* **Traceable Findings**: The engine generates human-readable reasons (e.g., *"Accessed sensitive service kms.amazonaws.com"*) attached directly to the `RiskScore`, forming the deterministic evidence required by the AI layer.
