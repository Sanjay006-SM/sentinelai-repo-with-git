# Stage 3: Enterprise Knowledge Graph Builder

This directory contains the Enterprise Knowledge Graph Builder, fulfilling **Stage 3** of the SentinelAI intelligence pipeline.

## 1. Problem Solved
Traditional relational databases (SQL) struggle to rapidly query multi-hop relationships. Identifying lateral movement in a cloud environment (e.g., Role A assumes Role B which accesses S3 Bucket C from IP D) requires a graph structure. 

Instead of storing cybersecurity data as isolated records in relational tables, this stage models every enterprise object as a graph node and connects related entities through semantic relationships. The Knowledge Graph Builder constructs this "Digital Twin" of the cloud environment in real-time.

## 2. Architecture
The feature communicates natively with a **Neo4j** graph database to construct the organizational twin.
- `GraphBuilder` (`graph_builder.py`): The abstraction layer wrapping the graph transactions.
- `Neo4j Queries` (`neo4j_queries.py`): Raw parameterized Cypher statements used to securely build and query the graph.

## 3. Execution Pipeline (Create Nodes → Create Relationships → Build Digital Twin)
**Pipeline Step 1: Entity Merging (Create Nodes)**
`MERGE` operations are executed for `Identity`, `Resource`, and `IPAddress` nodes, guaranteeing idempotency. Standardized enterprise entities generated by the Normalization Engine are converted into these nodes.

**Pipeline Step 2: Semantic Linking (Create Relationships)**
Based on normalized logs, the engine links nodes dynamically using `ON CREATE SET` and `ON MATCH SET` to track `count`, `first_seen`, and `last_seen` properties.

**Pipeline Step 3: Graph Publication (Build Digital Twin)**
The validated graph updates are committed and the latest Enterprise Knowledge Graph is made available to the Risk Intelligence Engine (Stage 4).

## 4. Major Features
- **Identity Triangulation**: Links identity nodes to originating IP addresses (`ORIGINATED_FROM`) to detect geographic anomalies.
- **Privilege Escalation Mapping**: Constructs `ASSUMED_ROLE` relationships to easily map identity chaining.
- **Resource Access Graphs**: Builds `ACCESSED_RESOURCE` edges allowing for blast-radius calculation.
- **Attack Path Generation**: Supports queries like `GET_ATTACK_PATH` using variable length paths (`-[*1..3]->`) to quickly visualize multi-stage exposures.

## 5. Reliability & Resilience
- Cypher queries utilize parameterized inputs (`$workspace_id`, `$arn`), preventing Cypher Injection vulnerabilities.
- Idempotent `MERGE` statements guarantee that duplicate pipeline runs simply update the `last_seen` timestamp and `count` rather than corrupting the topology.
