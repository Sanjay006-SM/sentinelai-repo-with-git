# SecurityEvent Contract

The `SecurityEvent` schema is the foundational data contract generated by Stage 1 and consumed by all downstream stages (Stage 2 Threat Graph, Stage 3 AI, Stage 4 Analytics, etc). 
It serves as a unified abstraction layer, masking the proprietary schemas of AWS, Suricata, OpenVAS, and Wazuh.

## Schema Definition

This is treated as a **public API** for subsequent stages.

| Field Name | Type | Required | Description | Example Value |
|------------|------|----------|-------------|---------------|
| `event_id` | `str` | Yes | Unique UUID for the event. Used for deduplication. | `"e4b3-4f2a..."` |
| `timestamp` | `datetime` | Yes | Standardized UTC timestamp of when the event occurred. | `"2026-07-15T12:00:00Z"` |
| `source` | `str` | Yes | Identifier of the originating connector. | `"wazuh"`, `"aws_cloudtrail"` |
| `event_type` | `str` | Yes | High-level categorization. Used for routing. | `"authentication_failed"` |
| `severity` | `float` | Yes | Normalized severity score (0.0 to 10.0). | `7.5` |
| `action` | `str` | Yes | Action attempted or executed. | `"ConsoleLogin"`, `"PortScan"` |
| `status` | `str` | Yes | Outcome of the action. | `"Success"`, `"Failure"` |
| `actor` | `Dict[str, Any]` | No | The entity performing the action (IP, username, ARN). | `{"type": "IP", "value": "192.168.1.5"}` |
| `resource` | `Dict[str, Any]` | No | The entity targeted by the action (Host, DB, S3 bucket). | `{"type": "EC2", "arn": "arn:aws:..."}` |
| `raw_data` | `Dict[str, Any]` | Yes | The original unmodified JSON from the source for forensics. | `{"rule": {"id": "5710"}}` |
| `metadata` | `Dict[str, Any]` | No | Enrichment data added by the pipeline (e.g. processing time). | `{"ingested_at": "..."}` |

## Extension Guidelines for Stages 2-8

1. **Never parse `raw_data` directly** if the data can be normalized into `actor` or `resource`. `raw_data` exists only for display/forensics, not for querying.
2. **Always respect `severity` scaling**. It is guaranteed to be mapped accurately from the source (e.g., CVSS 0-10, Wazuh 0-15 mapped to 0-10).
3. **Use `event_id`** as the primary key if storing events in a timeseries DB or relational database. Stage 1 guarantees deduplication within a time window, but idempotent inserts are still recommended downstream.
